\section{Criticità della soluzione e sviluppi futuri}

\subsection{Problemi incontrati durante lo sviluppo}
H2 persiste su disco tra un avvio e l'altro, quindi le modifiche allo schema non venivano applicate perché \texttt{CREATE TABLE IF NOT EXISTS} lasciava le tabelle esistenti. Abbiamo risolto aggiungendo \texttt{DROP TABLE} e \texttt{DELETE} all'inizio di \texttt{schema.sql} e \texttt{data.sql}. Secondo problema: Spring Security mantiene il ruolo nella sessione corrente, quindi aggiornare \texttt{AUTHORITIES} dopo un upgrade non ha effetto immediato; l'utente deve fare logout e login per attivarlo.

\subsection{Limiti della soluzione attuale}

\begin{itemize}
  \item L'amministratore non ha un'interfaccia per aggiungere o modificare programmi predefiniti. Per farlo è necessario modificare i file SQL del REST manipolandone li database.

  \item Le statistiche sono contatori cumulativi, non log. Non ci sono timestamp per sessione, quindi non è possibile analizzare la frequenza degli allenamenti nel tempo.

  \item Non abbiamo integrato un gateway di pagamento reale né implementato i requisiti GDPR.

  \item Il form dei contatti non invia nulla al centro.

  \item H2 embedded non è adatto alla produzione: poche connessioni concorrenti e persistenza legata al filesystem locale.
\end{itemize}

\subsection{Possibili miglioramenti futuri}

\begin{itemize}
  \item Sostituire H2 con PostgreSQL o MySQL.

  % \item Introdurre Redis per le sessioni, con invalidazione automatica dopo un upgrade.

  \item Collegare un gateway di pagamento e implementare la conformità GDPR.

  \item Aggiungere timestamp alle sessioni completate per abilitare statistiche temporali.
\end{itemize}
